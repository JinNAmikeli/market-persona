# 金融数字人 Agent Runtime 设计方法论

本文档用于把销售数字人中的 Sales Harness Agent 经验迁移到 `market_radar` 项目，形成金融数字人智能体的设计底座。

核心判断：金融数字人不应被设计成“行情问答机器人”或“荐股助手”，而应被设计成一个可追踪、可校验、可配置、可持续进化的市场陪伴 Runtime。

## 1. 先重定义业务问题

金融场景的核心问题不是“用户问什么，模型答什么”，也不是“根据行情给出买卖建议”。更准确的问题是：

- 用户面对市场信息过载，无法判断哪些变化重要。
- 用户容易被短期热点、社区情绪和个股波动牵着走。
- 用户缺少证据链，不知道一个主题为什么热、热度是否拥挤、风险在哪里。
- 用户无法持续跟踪自选股、关注主题和上一次复盘提出的观察点。
- 用户缺少适合自己知识水平的解释和复盘纪律。

因此，金融数字人的业务定位应是：

```text
市场观察员 + 投资教育老师 + 风险提醒员 + 自选股陪伴型复盘助手
```

它要回答四类问题：

1. 发生了什么？
2. 证据是什么？
3. 风险在哪里？
4. 下一步观察什么？

它不应直接输出买卖指令、收益预测或确定性判断。

## 2. 固定 SOP 与动态路径

销售场景中，首购更接近固定 SOP，复购更接近“主线目标 + 动态路径”。金融场景也类似：

```text
一次性市场查询：固定 SOP
长期市场陪伴：主线目标 + 动态路径
```

固定 SOP 示例：

- “今天指数涨跌如何？”
- “帮我生成一段收盘复盘。”
- “解释一下什么是拥挤度。”

动态路径示例：

- 用户连续关注 AI 硬件，需要跟踪主线是否扩散、是否退潮。
- 用户添加自选股后，需要跨天观察是否进入热榜、是否被热门帖子提到。
- 用户多次问同一类基础概念，说明需要降低解释难度。
- 用户倾向于追热点，需要更强的风险提醒和反事实提示。

只要进入动态路径，就不能只靠 prompt。系统必须有状态、记忆、规划、工具调用、校验和 trace。

## 3. 金融版 Harness Runtime

建议把金融数字人的底座命名为 `Market Harness Agent`。

它不是某一个业务流程，而是一套可复用 Runtime：

```text
Input
  用户问题 / 定时触发 / 自选股异动 / 市场数据刷新

State Reader
  用户画像 / 自选股 / 关注行业 / 历史对话 / 市场快照 / 上次观察清单

Planning
  判断本轮任务类型、推进目标、需要调用的工具和知识

Knowledge / Wiki
  主题页、术语解释、行业资料、风险清单、合规边界、证据来源

Tool Execution
  读取雪球雷达、历史趋势、热门帖、自选股状态、新闻公告、财报日历

Execution
  生成带证据的回答、复盘稿、口播稿、观察清单或风险提醒

Reflection
  校验事实、证据、合规边界、表达清晰度和是否越界荐股

Memory
  更新用户关注点、知识水平、阶段状态、阻塞点和下一轮观察重点

Trace
  记录触发原因、读取状态、规划判断、工具调用、证据、校验和记忆写回
```

这套 Runtime 应保持行业无关、业务可配置。`market_radar` 只是第一个 A 股市场观察场景。

## 4. Planning / Execution / Reflection / Memory 分工

### Planning

Planning 决定“这一轮往哪里推进”，不直接生成最终回复。

它应输出结构化计划，例如：

```json
{
  "task_type": "theme_explanation",
  "user_intent": "解释 AI 硬件为什么是当前主线",
  "required_state": ["latest_radar", "history_tail", "user_watchlist"],
  "required_tools": ["get_market_snapshot", "get_theme_signals"],
  "required_knowledge": ["theme:AI硬件/光通信", "risk:theme_crowding"],
  "response_goal": "解释主线证据、拥挤度和下一步观察指标",
  "forbidden_outputs": ["买入建议", "确定收益", "目标价"]
}
```

### Execution

Execution 根据 Planning 的结果生成最终内容。

金融场景中的输出应遵循：

- 先结论，后证据。
- 用数据和知识资产支撑判断。
- 用“观察指标”替代“操作建议”。
- 涉及个股时说明来源、时效和风险。
- 对小白用户解释专业词。

### Reflection

Reflection/Factuality 负责质量和边界校验。

最少校验四件事：

- 事实是否来自当前数据、历史数据或 Wiki 证据。
- 是否出现“可以买、必须卖、稳赚、一定涨”等越界表达。
- 是否混淆了热度、基本面、资金流、情绪等概念。
- 是否给出了可执行的下一步观察清单。

### Memory

Memory 不只是聊天上下文，而是用户旅程状态。

建议维护：

- 用户关注主题。
- 用户自选股。
- 用户知识水平。
- 高频问题。
- 历史提醒过的风险。
- 上一次复盘的观察清单。
- 本轮新增的关注点或误区。

## 5. 用 LLM-Wiki 替代松散 RAG

不要只把资料切片后丢进向量库。金融资料应整理成结构化知识资产。

建议的 Wiki 页面结构：

```text
topic_id: theme_ai_optical
title: AI硬件/光通信
version: 2026-07-06
status: reviewed
permission: public
tags: [A股, 主题投资, AI硬件, 光模块]
applicable_stage: [市场复盘, 主题解释, 小白教学]
forbidden_use: [个股买卖建议, 目标价预测]

sections:
  - title: 产业链位置
    content: ...
    evidence: ...

  - title: 热度驱动因素
    content: ...
    evidence: ...

  - title: 常见观察指标
    content: ...
    evidence: ...

  - title: 拥挤风险
    content: ...
    evidence: ...

  - title: 小白解释
    content: ...
    evidence: ...
```

每个 section 都应包含：

- 标题。
- 正文。
- 来源或证据。
- 适用场景。
- 禁用边界。
- 更新时间。
- 审核状态。

这样 Agent 的回答可以被追溯、审计和持续改进。

## 6. 证据链回复

金融数字人的回复不应只给“看法”，而应保留证据链。

推荐执行链路：

```text
1. 判断是否需要知识
2. 将用户问题补全为检索 query
3. 检索 TopK Wiki section
4. 读取实时市场数据和历史快照
5. 带证据生成回复
6. 交给 Factuality / Compliance 校验
7. 写入 Runtime trace
```

Trace 至少记录：

- 用户输入或触发条件。
- 读取了哪些状态。
- Planning 的任务判断。
- 调用了哪些工具。
- 检索 query。
- 命中的知识 section 和分数。
- 使用的市场数据快照。
- 最终回复。
- Reflection 结果。
- Memory 写回内容。

## 7. 主动触达与被动回复统一

金融数字人应同时支持三种模式：

```text
Passive
  用户主动提问，例如“今天市场怎么样？”

Activate
  系统主动触发，例如收盘后生成复盘、用户自选股进入热榜。

Custom Flow
  自定义流程，例如每日早盘观察、午盘更新、收盘复盘、周末主题复盘。
```

三种模式都应复用同一套 Market Harness Runtime，而不是各写一套 prompt。

示例触发：

- 市场数据刷新完成。
- 情绪分显著上升或下降。
- 某主题连续多次进入热榜。
- 用户自选股进入热门榜或被热帖提及。
- 某主题拥挤度过高。
- 到达固定时间点，需要生成口播稿。

## 8. 治理与人机协同

金融场景不适合完全自动自我修改。

推荐机制：

```text
自动发现问题 -> 自动生成修改建议 -> 人工确认 -> 上线配置
```

可以自动分析：

- 哪类问题回答失败率高。
- 哪些 Wiki 页面缺少证据。
- 哪些回答被 Reflection 拦截。
- 哪些用户问题需要新增工具。
- 哪些 prompt 或规则产生歧义。

但以下内容上线前必须人工确认：

- 合规边界调整。
- Wiki 内容更新。
- 投资相关表达模板。
- 主动触达策略。
- 用户画像标签规则。

## 9. 评估与对抗演练

可以仿照销售数字人的 CA/SA 多轮模拟，建立金融版模拟：

```text
UA: User Agent，模拟不同类型用户
FA: Finance Agent，被测金融数字人
RA: Reviewer Agent，评估事实、证据、合规和解释质量
```

用户类型示例：

- 小白型：概念不清，容易被术语吓退。
- 追热点型：频繁问“能不能买”。
- 焦虑型：自选股下跌后寻找确定答案。
- 研究型：要求证据、历史对比和行业逻辑。
- 内容创作者型：需要口播稿和结构化素材。

评估维度：

- 是否准确识别意图。
- 是否调用正确工具和知识。
- 是否给出证据链。
- 是否避免荐股和确定性收益表达。
- 是否适合用户知识水平。
- 是否给出下一步观察指标。
- 是否正确更新 Memory。

## 10. `market_radar` 的 MVP 落地路线

当前 `market_radar` 已有能力：

- 雪球指数、热榜、关注榜、热门帖采集。
- 历史快照。
- 前端主线识别、情绪分、拥挤度。
- DeepSeek 单次复盘。
- 自选股观察台。

建议下一阶段只做最小闭环：

```text
Planning 输出下一步动作
Execution 带证据回复
Memory 回写用户状态
Trace 留下运行记录
```

优先级建议：

1. 统一数据目录，避免根目录 `data/` 和 `market_radar/data/` 产生歧义。
2. 将主线识别、情绪分、拥挤度从前端迁移到后端 `market_signals`。
3. 新增 `/api/agent/chat`，支持用户自然语言提问。
4. 新增最小 Wiki 目录，例如 `market_radar/wiki/`。
5. 新增 Runtime trace，例如 `market_radar/data/agent_traces.jsonl`。
6. 新增 Memory，例如 `market_radar/data/user_memory.json`。
7. 在现有 DeepSeek 调用前后加入 Planning 和 Reflection。
8. 再考虑 TTS、数字人头像和口播表现层。

第一版目标不应是“金融数字人全量系统”，而是：

```text
每天帮用户解释市场主线、跟踪自选股，并生成带证据的风险观察清单。
```

## 11. 边界原则

金融数字人必须内置边界，而不是依赖模型临场自觉。

允许：

- 市场观察。
- 主题解释。
- 风险提示。
- 自选股热度跟踪。
- 财报、公告、新闻的事实梳理。
- 投资知识教育。
- 生成复盘稿和口播稿。

不允许：

- 直接买入或卖出建议。
- 确定性收益预测。
- 目标价承诺。
- 替用户做投资决策。
- 使用无证据来源的结论冒充事实。
- 忽略数据时效。

推荐转写：

```text
我不能给出买卖指令，但可以帮你拆解这个标的/板块当前的热度来源、风险点和后续观察指标。
```

## 12. 后续可沉淀为 Skill

当这套方法在 `market_radar` 中跑通后，可以沉淀成 Codex skill，例如：

```text
agent-runtime-designer
business-agent-architect
market-harness-agent
```

Skill 的职责不是保存这篇文档，而是把它变成可重复执行的工作流：

- 输入一个业务场景。
- 重定义业务问题。
- 判断固定 SOP 还是动态路径。
- 设计 Runtime。
- 设计 Planning / Execution / Reflection / Memory。
- 设计 Wiki 和 Trace。
- 输出 MVP 落地路线。

在那之前，本文件作为金融数字人方案设计的第一版方法论基线。
